1. RESUMEN EJECUTIVO
============================================================

SASU / Carnets UAGro Pro es un sistema de atencion en salud
universitaria construido con una arquitectura offline-first. La aplicacion
principal esta desarrollada en Flutter y utiliza una base local SQLite
llamada cres_carnets.sqlite. La base local permite operar aun cuando no
exista conexion a internet o cuando el backend no este disponible.

El sistema sincroniza informacion con un backend FastAPI desplegado en
Render. El backend guarda informacion en Azure Cosmos DB. GitHub y GitHub
Releases se utilizan como repositorio, historico de versiones, canal de
distribucion y soporte para actualizaciones.

La principal fortaleza del sistema es que puede seguir capturando datos
localmente. El principal riesgo operativo es perder la base
cres_carnets.sqlite antes de sincronizar datos pendientes. Todo dato con
synced=false debe considerarse dato en riesgo hasta que sea confirmado en
la nube.

El blindaje institucional debe concentrarse en:

- Respaldos diarios de cres_carnets.sqlite.
- Respaldos semanales del proyecto completo.
- Custodia segura de credenciales y variables de entorno.
- Documentacion de Render, Azure Cosmos DB y GitHub.
- Prueba periodica de recuperacion.
- Separacion y control de scripts peligrosos.
- Definicion institucional de responsables.

2. ARQUITECTURA GENERAL
============================================================

Arquitectura resumida:

Usuario SASU
  -> App Flutter
    -> SQLite local: cres_carnets.sqlite
    -> ApiService / AuthService / SyncService
      -> Backend FastAPI en Render
        -> Azure Cosmos DB
          -> Contenedores: carnets, notas, citas_id, tarjeta_vacunacion,
             promociones_salud, usuarios, auditoria

Componentes principales:

1. Flutter
   Aplicacion de escritorio/movil. Contiene pantallas de login,
   dashboard, carnets, notas, citas, vacunacion, promocion de salud,
   diagnostico, limpieza y reportes.

2. SQLite local
   Base de datos local ubicada normalmente en:
   C:\Users\[Usuario]\Documents\cres_carnets.sqlite

3. FastAPI
   Backend Python ubicado en temp_backend. Expone endpoints de auth,
   carnets, notas, citas, vacunacion, promociones, auditoria y updates.

4. Render
   Plataforma donde se despliega el backend FastAPI.

5. Azure Cosmos DB
   Base remota donde se almacenan los datos sincronizados.

6. GitHub
   Repositorio de codigo, workflows y releases.

7. GitHub Releases
   Canal para distribucion de instaladores y actualizaciones.

4. DIAGRAMAS TEXTUALES
============================================================

4.1 Arquitectura general

Usuario
  -> Flutter
    -> SQLite local
    -> HTTP API
      -> FastAPI / Render
        -> Azure Cosmos DB

4.2 Flujo offline-first

Usuario captura dato
  -> App valida campos
  -> Guarda en SQLite
  -> Registro inicia con synced=false
  -> Si hay conexion, intenta sincronizar
  -> Si la nube confirma, synced=true
  -> Si falla, permanece pendiente

4.3 Flujo de sincronizacion

SyncService revisa pendientes
  -> HealthRecords / Notes / Citas / VacunacionesPendientes
  -> ApiService envia a FastAPI
  -> FastAPI guarda en Cosmos DB
  -> Respuesta exitosa
  -> SQLite marca synced=true

4.4 Flujo de recuperacion

Falla o perdida
  -> Activar Recovery Kit
  -> Restaurar proyecto/release
  -> Restaurar SQLite
  -> Verificar Render
  -> Verificar Azure Cosmos DB
  -> Verificar GitHub
  -> Probar login
  -> Probar captura
  -> Probar sincronizacion
  -> Reanudar operacion

4.5 Flujo de respaldo

Cierre operativo
  -> Verificar pendientes
  -> Cerrar app
  -> Copiar cres_carnets.sqlite
  -> Guardar copia fechada
  -> Respaldar proyecto/release
  -> Registrar evidencia

11. SINCRONIZACION OFFLINE-FIRST
============================================================

Reglas:

- synced=false: Registro pendiente local, aun no confirmado por nube.
- synced=true: Registro ya sincronizado exitosamente.

Funciones principales:

- SyncService.syncAll()
- SyncService.syncRecord()
- SyncService.syncNote()
- ApiService.pushSingleCarnet()
- ApiService.pushSingleNote()
- ApiService.createCita()
- ApiService.createVacunacion()
- syncVacunacionesPendientes()

Pantallas para pendientes:
- PendingSyncScreen
- Dashboard
- DatabaseCleanerScreen (con precaucion)

Riesgo:
Si se pierde cres_carnets.sqlite antes de sincronizar, los registros con
synced=false pueden perderse definitivamente.

============================================================
12. SEGURIDAD, ROLES Y PERMISOS
============================================================

JWT:
El backend usa tokens JWT con algoritmo HS256 y expiracion aproximada de
8 horas. Flutter guarda el token en FlutterSecureStorage.

Elementos guardados localmente:
- auth_token
- auth_user
- cached_password
- hashes/cache offline

Roles detectados:
- admin
- medico
- nutricion
- psicologia
- odontologia
- enfermeria
- recepcion
- servicios_estudiantiles
- lectura

Riesgos de seguridad:
- cached_password en almacenamiento local seguro.
- AuthGate/PIN deshabilitado temporalmente.
- Archivos sensibles dentro del arbol del proyecto.
- Diferencias entre permisos frontend y backend.
- Variables de entorno inconsistentes COSMOS_URL / COSMOS_ENDPOINT.

17. PLAN DE CONTINUIDAD OPERATIVA
============================================================

Escenario: Render se cae
Impacto:
- No hay login online.
- No hay sincronizacion.
- No hay consultas nube.
- No hay panel admin.
Sigue funcionando:
- Captura local de carnets/notas.
- SQLite.
- PDFs locales.
Accion:
- Operar offline.
- No limpiar datos.
- Verificar pendientes cuando vuelva.

Escenario: Azure Cosmos DB falla
Impacto:
- Backend puede responder, pero no guardar/consultar datos remotos.
- Login puede fallar si usuarios dependen de Cosmos.
Accion:
- Mantener captura local.
- Revisar Azure Portal.
- No recrear contenedores sin respaldo.

Escenario: Se pierde cres_carnets.sqlite
Impacto:
- Perdida de datos locales.
- Perdida de registros synced=false.
Accion:
- Restaurar ultimo backup.
- Consultar nube para lo ya sincronizado.

Escenario: GitHub desaparece o se pierde acceso
Impacto:
- Sin repo remoto.
- Sin releases.
- Sin workflows.
Accion:
- Usar copia local y releases externas.
- Restaurar repo desde backup.

Escenario: Se pierde computadora principal
Impacto:
- Se pierde entorno local y posiblemente SQLite.
Accion:
- Activar Recovery Kit.
- Restaurar equipo, proyecto, SQLite y accesos.

19. PROCEDIMIENTOS NORMALIZADOS DE OPERACION
============================================================

PNO Alta de usuario
Objetivo: Crear usuario con rol y campus correctos.
Responsable: Administrador SASU.
Procedimiento:
1. Recibir solicitud formal.
2. Validar rol y campus.
3. Crear usuario.
4. Asignar permisos minimos.
5. Registrar alta.
Evidencia: Registro de alta.

PNO Baja de usuario
Objetivo: Retirar accesos.
Responsable: Administrador SASU.
Procedimiento:
1. Recibir solicitud.
2. Identificar usuario.
3. Desactivar cuenta.
4. Registrar baja.
Evidencia: Registro de baja.

PNO Respaldo diario
Objetivo: Proteger SQLite.
Responsable: Operador local.
Procedimiento:
1. Cerrar app.
2. Copiar cres_carnets.sqlite.
3. Guardar con fecha.
4. Verificar tamano.
5. Registrar evidencia.

PNO Respaldo semanal
Objetivo: Proteger sistema completo.
Responsable: Tecnico designado.
Procedimiento:
1. Copiar proyecto.
2. Copiar release estable.
3. Copiar documentacion.
4. Verificar SQLite reciente.
5. Registrar evidencia.

PNO Recuperacion de SQLite
1. Cerrar app.
2. Ubicar backup valido.
3. Copiar a Documents.
4. Nombrar cres_carnets.sqlite.
5. Abrir app.
6. Verificar expedientes.

PNO Recuperacion completa
1. Preparar equipo.
2. Restaurar Recovery Kit.
3. Instalar release.
4. Restaurar SQLite.
5. Verificar GitHub, Render y Azure.
6. Probar login.
7. Probar captura.
8. Probar sincronizacion.

PNO Verificacion de sincronizacion
1. Abrir pendientes.
2. Identificar synced=false.
3. Sincronizar.
4. Verificar resultados.
5. No limpiar si hay errores.

PNO Atencion de incidentes
1. Clasificar incidente.
2. Detener acciones peligrosas.
3. Respaldar estado actual.
4. Revisar servicio afectado.
5. Informar responsables.
6. Documentar.

- Sin indices explicitos en SQLite.

- Codigo legacy Cloudant.
- Pantallas con logica grande.

Bajos:
- Encoding roto en algunos textos.
- Documentacion dispersa.
- Backups manuales no estandarizados.

28. GLOSARIO TECNICO Y OPERATIVO
============================================================

SASU:
Sistema de Atencion en Salud Universitaria.

SQLite:
Base de datos local usada por la app para operar offline.

cres_carnets.sqlite:
Archivo fisico de SQLite que contiene datos locales.

Render:
Servicio en nube donde se despliega FastAPI.

Azure Cosmos DB:
Base de datos remota donde se guardan datos sincronizados.

FastAPI:
Framework Python usado para construir el backend.

GitHub:
Plataforma de codigo, historial y automatizacion.

GitHub Releases:
Canal de publicacion de instaladores y actualizaciones.

Observatorio:
Funcion institucional para analizar indicadores agregados de salud.

synced=true:
Registro local confirmado en nube.

synced=false:
Registro pendiente local. Debe considerarse dato en riesgo.

Recovery Kit:
Paquete de respaldo, documentacion y procedimientos para restaurar SASU.

Continuidad Operativa:
Capacidad de seguir operando o recuperarse despues de incidentes.

RTO:
Tiempo objetivo de recuperacion.

RPO:
Perdida maxima aceptable de datos medida desde el ultimo respaldo.
